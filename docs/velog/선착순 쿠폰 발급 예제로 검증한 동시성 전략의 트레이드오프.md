> 이 글은 실제 운영 장애 회고가 아니라, 학습용 선착순 쿠폰 발급 예제를 기반으로 동시성 전략을 비교/검증한 내용입니다.
> 트래픽과 장애 사례도 운영 데이터가 아닌 재현 실험 시나리오를 기준으로 합니다.

이번 글의 결론부터 먼저 말씀드리면 다음과 같습니다.

선착순 쿠폰 발급에서 동시성 전략 선택은 "어떤 락이 더 고급인가"의 문제가 아니었습니다. 실제로는 **어떤 경계를 트랜잭션으로 보장할지**, **실패를 어디까지 정상 시나리오로 받아들일지**를 정하는 문제였습니다.

같은 예제 유스케이스를 6가지 전략으로 구현해 보니, 성능보다 먼저 드러난 것은 정합성 경계와 설계 복잡도였습니다.

## 문제 정의: 예제에서 지켜야 했던 것은 두 가지 불변식

유스케이스는 단순합니다. 학습용 선착순 쿠폰 발급 예제입니다.

아래 두 조건이 깨지면 구현은 실패입니다.

1. 총 발급 수량이 한도를 넘지 않는다
2. 동일 사용자가 중복 발급되지 않는다

비교 전략 6가지는 다음과 같습니다.

- `NO_LOCK`: 락 없이 처리하며 경합 시 정합성 깨짐을 기준선으로 보는 전략이다
- `SYNCHRONIZED`: JVM `synchronized`로 임계 구역을 보호하는 인메모리 락 전략이다
- `REENTRANT_LOCK`: `ReentrantLock`으로 락 획득/해제를 명시적으로 제어하고, 락을 즉시 얻지 못하면 실패를 반환하는 인메모리 락 전략이다
- `OPTIMISTIC_LOCK`: 충돌이 낮다는 가정으로 버전 충돌 시 재시도하는 낙관적 잠금 전략이다
- `PESSIMISTIC_LOCK`: 충돌 가능성을 전제로 DB 행 잠금을 먼저 획득해 처리하는 비관적 잠금 전략이다
- `REDIS_SORTED_SET`: Redis에 선착순 예약 순서를 기록하고 DB에서 최종 확정하는 2단계 전략이다

핵심은 락 자체가 아니라, 각 전략이 위 두 불변식을 어떤 비용으로 지키는지 확인하는 것이었습니다.

## 왜 전략 패턴으로 시작했는가

먼저 고정한 것은 전략 구현이 아니라 비교 경계였습니다.

- `CouponIssueStrategy`: 전략 인터페이스
- `CouponIssueStrategyRouter`: 전략 선택
- `CouponIssueService`: 동일 진입점

이 구조를 먼저 고정해두면 전략을 바꿔도 입력/출력 계약은 유지됩니다. 즉, 전략 비교가 구현 취향 비교로 흐르지 않습니다.

실제 진입점 코드는 아래처럼 매우 얇습니다.

```java
@Service
@RequiredArgsConstructor
public class CouponIssueService {

    private final CouponIssueStrategyRouter strategyRouter;

    public IssueResult issue(CouponIssueStrategyType strategyType, CouponIssueCommand command) {
        return strategyRouter.get(strategyType).issue(command);
    }
}
```

## 구현에서 먼저 깨진 지점들

### 1) 예외를 응답으로 바꾸는 것만으로는 안전하지 않았습니다

중복 발급 예외를 `IssueResult`로 바꿔도, 트랜잭션이 rollback-only가 된 상태에서는 커밋 시점 예외가 다시 발생할 수 있었습니다.

수정은 아래처럼 진행했습니다.

- 전략 메서드의 선언형 `@Transactional` 제거
- `TransactionTemplate` 기반으로 경계 명시
- 공통 처리기(`CouponIssueProcessor`)는 `Propagation.MANDATORY` 강제

핵심은 예외 메시지 변환이 아니라 **트랜잭션 수명주기 통제**였습니다.

### 2) Redis에서 "품절" 판단이 오탐이 될 수 있었습니다

초기 구현은 ZSET 크기를 기준으로 즉시 `SOLD_OUT`을 반환할 수 있었습니다. 하지만 stale reservation이 남아 있으면, 실제로는 수량이 남아 있어도 품절로 오판할 수 있습니다.

수정은 아래처럼 진행했습니다.

- `zCard >= limit` 즉시 품절 판단 제거
- 예약 충돌 시 DB 이력으로 교차 검증
- stale reservation 정리 후 제한 횟수 재시도

Redis 상태를 단독 진실로 보지 않도록 경계를 조정했습니다.

### 3) 선착순 점수에 시간값만 쓰면 순서 충돌이 생깁니다

`currentTimeMillis()` 기반 점수는 동일 ms 요청에서 동점이 생깁니다.

수정:

- Redis `INCR` 시퀀스를 점수로 사용

완전한 전역 순서를 보장하는 설계는 아니지만, 동일 키 기준 단조 증가를 확보했습니다.

### 4) 낙관적 락 재시도는 backoff가 없으면 쉽게 폭주합니다

바쁜 대기 기반 재시도는 경합이 커질수록 DB와 CPU를 함께 밀어붙입니다.

수정:

- 짧은 exponential backoff + jitter 적용

재시도 유지와 시스템 압력 완화를 동시에 노렸습니다.

### 5) 실험 환경 설정 하나가 결론을 왜곡할 수 있습니다

실험 대상 DB를 MySQL로 두었는데 dialect를 H2로 고정하면 락 동작 검증 신뢰도가 떨어집니다.

수정:

- 메인 설정: dialect 강제 제거
- 테스트 설정: H2를 테스트 프로필에서만 명시

동시성 실험에서는 코드만큼 환경 경계가 중요했습니다.

## 역할 분리: 무엇을 공통으로 두고 무엇을 전략으로 남겼는가

### 공통 처리기 (`CouponIssueProcessor`)

- 중복 사용자 확인
- 수량 차감
- 발급 이력 저장

### 전략 구현체

- 락/조회/예약 방식
- 재시도 정책
- 실패 코드 노출 방식

이 분리가 없으면 전략 비교가 아니라 코드 중복 비교가 됩니다. 분리 이후에야 전략별 실패 모델 차이가 명확해졌습니다.

### 공통 처리기 코드

아래 코드는 "발급 불변식"을 한 곳으로 모은 핵심입니다.

```java
@Service
@RequiredArgsConstructor
public class CouponIssueProcessor {

    private final CouponIssueHistoryRepository issueHistoryRepository;

    @Transactional(propagation = Propagation.MANDATORY)
    public IssueResult issue(Coupon coupon, Long userId) {
        if (issueHistoryRepository.existsByCouponIdAndUserId(coupon.getId(), userId)) {
            return IssueResult.failure("ALREADY_ISSUED", "이미 발급받은 사용자입니다.");
        }
        if (!coupon.tryIssue()) {
            return IssueResult.failure("SOLD_OUT", "쿠폰이 모두 소진되었습니다.");
        }
        issueHistoryRepository.saveAndFlush(new CouponIssueHistory(coupon.getId(), userId));
        return IssueResult.success("쿠폰 발급에 성공했습니다.");
    }
}
```

전략은 이 공통 처리기를 호출하기 전에 "락/재시도/예약" 정책만 결정합니다.

```java
@Override
public IssueResult issue(CouponIssueCommand command) {
    IssueResult result = transactionTemplate.execute(status -> issueOnce(command));
    if (result == null) {
        return IssueResult.failure("TRANSACTION_EXECUTION_FAILED", "트랜잭션 실행 결과가 없습니다.");
    }
    return result;
}
```

## 검증: 설명 대신 동시성 테스트로 확인한 것

추가한 통합 테스트 시나리오는 아래 두 가지입니다.

1. 서로 다른 사용자 200개 동시 요청: 목표는 총 발급량 불변식 확인

2. 동일 사용자 100개 동시 요청: 목표는 중복 발급 불변식 확인

현재 테스트는 DB 기반 전략(`NO_LOCK`, `SYNCHRONIZED`, `REENTRANT_LOCK`, `OPTIMISTIC_LOCK`, `PESSIMISTIC_LOCK`)을 우선 대상으로 두었습니다. Redis 전략은 외부 의존성과 복구 시나리오를 포함하므로 별도 테스트 트랙으로 분리하는 편이 적절하다고 판단했습니다.

테스트 코드는 아래처럼 파라미터 기반으로 같은 시나리오를 반복합니다.

```java
@ParameterizedTest
@EnumSource(
        value = CouponIssueStrategyType.class,
        names = {"SYNCHRONIZED", "REENTRANT_LOCK", "OPTIMISTIC_LOCK", "PESSIMISTIC_LOCK"}
)
void 동시_요청에서도_총_발급량을_초과하지_않는다(CouponIssueStrategyType strategyType) throws InterruptedException {
    Coupon coupon = couponRepository.saveAndFlush(Coupon.create("한정 쿠폰", 40));
    ExecutionResult executionResult = runConcurrentIssue(
            strategyType,
            coupon.getId(),
            200,
            index -> (long) (index + 1)
    );
    assertThat(executionResult.errors()).isEmpty();
}
```

## 전략별 트레이드오프 요약

### NO_LOCK

- 장점: 구현이 가장 단순하고 기준선으로 유용
- 단점: 경합 상황의 정합성 문제를 그대로 노출

### SYNCHRONIZED / REENTRANT_LOCK

- 장점: 단일 인스턴스에서 빠르게 안정화 가능
- 단점: 멀티 인스턴스 확장 시 락 공유 불가

### OPTIMISTIC_LOCK

- 장점: 충돌이 낮은 구간에서 효율적
- 단점: 경합이 높아지면 재시도 비용 급증

### PESSIMISTIC_LOCK

- 장점: 정합성 경계가 가장 직관적
- 단점: 락 대기와 경합 비용이 명확하게 발생

### REDIS_SORTED_SET

- 장점: 예약/선착순 제어 분리에 유리
- 단점: Redis-DB 이중 상태 정합성, stale 정리, 복구 설계 필요

## 선택 기준: 어떤 전략을 언제 택할 것인가

이번 구현 기준에서 정리한 선택 가이드는 아래와 같습니다.

1. 학습/단일 인스턴스 환경: `SYNCHRONIZED`, `REENTRANT_LOCK`
2. 정합성 최우선: `PESSIMISTIC_LOCK`
3. 충돌이 낮은 쓰기 패턴: `OPTIMISTIC_LOCK`
4. 선착순 예약 분리가 필요한 경우: `REDIS_SORTED_SET`

단, 도입 전에 복구 시나리오를 먼저 설계해야 합니다.

전략은 "더 빠른 락"을 찾는 도구가 아니라, **실패 비용을 어떤 계층에서 감당할지 정하는 도구**였습니다.

## 정리

이번 작업에서 얻은 결론은 4가지입니다.

1. 전략 패턴의 핵심 가치는 확장성이 아니라 비교 경계 고정이다
2. 동시성 문제의 본질은 락 API가 아니라 트랜잭션 경계와 실패 모델이다
3. Redis를 넣는 순간 성능 최적화 문제가 아니라 이중 저장소 일관성 관리 문제가 시작된다
4. 동시성 구현은 설명으로 완성되지 않으며, 동시성 테스트가 있어야 신뢰할 수 있다

한 줄로 요약하면 다음과 같습니다.

"동시성 전략 선택은 락 선택이 아니라 정합성 경계 설계입니다."
