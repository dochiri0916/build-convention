# 기술 블로그 작성 레퍼런스 (Toss / MUSINSA 스타일)

## 목적
- 기술 소개보다 **문제 해결 의사결정 기록**에 초점을 맞춘다.
- "무엇을 했다"보다 "왜 이 선택을 했는지"를 재현 가능하게 남긴다.

## 공통 문맥 구조
1. 문제 배경: 운영에서 어떤 비용/위험이 있었는지
2. 관찰/재현: 로그, 지표, 재현 조건
3. 원인 추적: 가설과 검증 과정
4. 대안 비교: 후보별 장단점과 선택 기준
5. 적용 방식: 단계별 구현 변경
6. 결과: 전/후 지표
7. 트레이드오프: 남은 한계와 운영 비용
8. 정리: 적용 가능한 조건과 비적용 조건

## Toss 스타일 포인트
- "문제 -> 해결 -> 결과" 전개가 빠르고 명확하다.
- 수치/근거(지연 시간, 에러율, 처리량, 재현 조건)를 구체적으로 제시한다.
- 결론을 먼저 보여주고, 뒤에서 근거를 쌓는다.

## MUSINSA 스타일 포인트
- 문제를 정의한 뒤 해결 범위를 좁혀간다.
- 개선 단계를 분리해 점진적으로 설명한다.
- 기술 결정이 조직/운영 비용에 미치는 영향까지 함께 다룬다.

## 이번 쿠폰 동시성 글에 적용할 문맥
- "락 종류 설명" 중심이 아니라, 아래 불변식 중심으로 작성한다.
- 불변식 1: 총 발급 수량이 한도를 넘지 않음
- 불변식 2: 동일 사용자 중복 발급 없음
- 전략 패턴은 구현 기법이 아니라 **비교 가능한 실험 경계**로 설명한다.

## 권장 섹션 템플릿
1. 문제 정의: 선착순 쿠폰의 정합성 불변식
2. 설계 기준: 전략 패턴과 책임 경계
3. 전략별 가설과 위험
4. 실제 장애/결함과 수정 과정
5. 동시성 테스트와 검증 지표
6. 결과 비교와 트레이드오프
7. 언제 어떤 전략을 선택할지

## 작성 체크리스트
- 실제로 터진 문제와 수정 전후를 포함했는가
- 의사결정 기준이 코드/지표와 연결되는가
- 트레이드오프를 감추지 않았는가
- 적용 범위와 비적용 범위를 분리했는가
