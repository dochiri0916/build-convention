N+1 문제를 운영에서 늦게 발견하면, 원인 분석 비용은 반드시 기능 개발 비용보다 커집니다. 한 메서드에서 `0.1초` 느려지는 수준은 작아 보여도, 그런 구간이 여러 개 쌓이면 요청 지연은 쉽게 수 초 단위로 올라갈 수 있습니다.

문제는 N+1이 보통 "한 번에 크게" 터지지 않는다는 점입니다. 작은 누수가 여러 엔드포인트에 분산되어 누적되기 때문에, 장애가 생긴 뒤에야 존재를 확인하는 경우가 많습니다.

로그를 뒤져 원인을 찾을 수는 있습니다. 다만 그 방식은 변경마다 반복하기 어렵고, 무엇보다 회귀를 자동으로 막지 못합니다.

그래서 질문을 "N+1을 어떻게 고칠까"에서 "N+1성 쿼리 패턴이 들어왔을 때 어떻게 테스트에서 즉시 실패시키고, 어떻게 회귀를 차단할까"로 바꿨습니다.

---

## 쿼리 카운터에 필요했던 요건

필요했던 것은 운영 관측 도구가 아니라 테스트 단계의 회귀 차단 장치였습니다.

요건은 아래 4가지였습니다.

- 테스트 또는 메서드 단위로 SQL 집계를 분리할 수 있어야 한다
- N+1 핵심 신호인 반복 `SELECT`를 추적할 수 있어야 한다
- 임계치 초과 시 JUnit 실패로 즉시 노출되어야 한다
- Spring Data JPA 환경에서 도입 비용이 과하지 않아야 한다

핵심 목표는 "정확한 절대 진단"보다 **"빠른 회귀 감지"**였습니다.

---

## 왜 모듈화했는가

초기 구현에서 가장 먼저 부딪힌 문제는 "기능은 하나인데 코드가 관심사별로 섞이면 학습과 확장이 어려워진다"는 점이었습니다. 그래서 구현을 아래 4개 모듈로 나눴습니다.

- `core`: 쿼리 이벤트 모델, 정규화, fingerprint, 집계, 정책 검증
- `spring-boot-starter`: Hibernate `StatementInspector` 자동 등록과 Spring Boot 연동
- `test-support`: `@EnableQueryGuard`, `@QueryBudget`, `QueryGuardExtension`, `QueryGuardAssertions`
- `sample-app`: N+1 재현/해결 시나리오와 통합 테스트

이 구조의 장점은 명확했습니다.

- 수집 로직(`starter`)과 판단 로직(`core`)을 분리하면 테스트가 쉬워진다
- 서비스 적용 코드는 얇게 유지하고 내부 로직은 독립적으로 개선할 수 있다
- 글이나 면접에서 "수집 -> 정규화 -> 집계 -> 검증"을 모듈 단위로 설명하기 쉬워진다
---

## 대안 비교와 선택 기준

초기에는 아래 3가지를 후보로 봤습니다.

1. `hibernate.show_sql`/로그 파싱
2. `DataSource` 래핑 기반 JDBC 계측
3. Hibernate `StatementInspector`

결론부터 말씀드리면 1번은 "확인용"이지 "검증용"이 아니었습니다. `show_sql` 출력은 결국 문자열 로그입니다. 테스트 단위 경계를 안정적으로 나누기 어렵고, 포맷 의존성도 큽니다. 그래서 초기 버전의 공식 수집 경로로는 고려하지 않았습니다.

`DataSource` 래핑은 가장 범용적이었습니다. ORM 바깥의 JDBC 호출까지 폭넓게 관찰할 수 있기 때문입니다. 대신 `Connection`/`PreparedStatement` 프록시, 스프링 연동, 예외/리소스 처리까지 구현해야 해서 초기 버전 범위에서는 비용이 컸습니다.

반면 `StatementInspector`는 현재 스택과 가장 잘 맞았습니다.

- Hibernate Javadoc 기준 `StatementInspector`는 SQL을 JDBC statement 준비 전에 가로챈다
- N+1 회귀 탐지는 반복 `SELECT` 식별이 핵심이므로 이 지점 수집만으로도 실용적인 검증이 가능하다
- Spring Data JPA 중심 서비스에서는 충분히 유효하다
- Hibernate를 거치지 않는 raw JDBC/MyBatis 쿼리는 별도 계측이 필요하다

즉, 선택 기준은 "가장 강력한 방식"이 아니라 "현재 제약에서 회귀 차단을 가장 빨리 구축할 수 있는 방식"이었습니다.

## 원본 SQL이 어떻게 변환되는가

핵심은 "원본 SQL 문자열을 그대로 비교하지 않는다"는 점입니다. `normalizedSql`과 `fingerprint`를 거치면서 아래처럼 비교 가능한 형태로 바뀝니다.

### 예시 A

원본 SQL:

```sql
SELECT m.id, m.name
FROM members m
WHERE m.id IN (1, 2, 3)
  AND m.name = 'Neo';
```

`normalizedSql`:

```sql
select m.id, m.name from members m where m.id in (?) and m.name = ?
```

`fingerprint`:

```sql
select id, name from members where id in (?) and name = ?
```

변환 포인트:

- 주석/세미콜론 제거
- 숫자/문자 literal을 `?`로 치환
- 공백/대소문자 정규화
- alias(`m.`) 제거
### 예시 B (같은 패턴, 다른 SQL)

원본 SQL:

```sql
select m1_0.id, m1_0.name
from members m1_0
where m1_0.id in (7, 8, 9)
  and m1_0.name = 'Trinity'
```

`normalizedSql`:

```sql
select m1_0.id, m1_0.name from members m1_0 where m1_0.id in (?) and m1_0.name = ?
```

`fingerprint`:

```sql
select id, name from members where id in (?) and name = ?
```

결과적으로 A와 B는 원본 SQL은 다르지만 최종 `fingerprint`는 동일합니다. 그래서 "값만 다르고 형태가 같은 반복 SELECT"를 같은 패턴으로 집계할 수 있습니다. 다만 동일 `fingerprint`는 "동일 패턴" 신호이지, 곧바로 "문제 확정"을 의미하지는 않습니다.

---

## 구현 흐름: 수집, 정규화, 집계, 검증

구현은 아래 4단계로 나눴습니다.

1. SQL 수집: `StatementInspector.inspect(sql)`에서 SQL 이벤트를 받는다
2. SQL 정규화: literal/주석/공백 차이를 줄여 비교 가능한 형태로 통일한다
3. 패턴 집계: fingerprint 단위로 반복 `SELECT` 실행 횟수를 누적한다
4. 정책 검증: 쿼리 허용 기준을 넘으면 테스트를 실패시킨다

아래 코드는 구조 설명을 위한 최소 예시입니다. 실제 프로젝트 코드로 교체하더라도 "수집 -> 정규화 -> 집계 -> 검증" 흐름은 동일합니다.

`StatementInspector`를 Hibernate에 주입하는 설정:

```java
@Bean
public HibernatePropertiesCustomizer nPlusGuardHibernatePropertiesCustomizer(
        NPlusGuardStatementInspector statementInspector
) {
    return properties -> properties.put(
            "hibernate.session_factory.statement_inspector",
            statementInspector
    );
}
```

실제 수집 지점:

```java
@Override
public String inspect(String sql) {
    if (sql == null || sql.isBlank()) {
        return sql;
    }

    scopeManager.currentScope().ifPresent(scope -> scope.record(QueryEvent.capture(sql)));
    return sql;
}
```

여기서 `QueryEvent.capture(sql)`는 다음 정보를 만듭니다.

- `normalizedSql`: 값/포맷 차이를 줄인 비교용 SQL
- `fingerprint`: alias 차이까지 줄인 반복 패턴 키
- `queryType`: `SELECT`/`INSERT`/`UPDATE`/`DELETE` 분류

이렇게 분리해두면 "값만 다른 같은 쿼리"를 같은 패턴으로 계산할 수 있고, 반복 `SELECT` 상위 패턴을 기준으로 N+1성 회귀를 빠르게 드러낼 수 있습니다.

파이프라인을 한 줄로 요약하면 아래와 같습니다.

`inspect(sql)` 수집 -> `normalize(sql)` (주석/literal/공백 정리) -> `fingerprint(normalizedSql)` (alias 제거) -> `QuerySummary` 집계(반복 SELECT 계산) -> `GuardPolicy` 검사(허용 기준 초과 시 실패)

## 쿼리 허용 기준 테스트를 어떻게 설계했는가

회귀 차단은 결국 정책 설계 문제였습니다. `GuardPolicy`는 아래 3가지를 기준으로 검증합니다.

- `maxTotalQueries`: 전체 쿼리 수 상한
- `maxSelectQueries`: 조회 쿼리 수 상한
- `maxRepeatedSelectExecutions`: 동일 조회 패턴 반복 상한

테스트 사용 방식은 `@EnableQueryGuard + @QueryBudget` 조합으로 고정했습니다.

```java
@SpringBootTest
@EnableQueryGuard
class MemberQueryTest {

    @Test
    @QueryBudget(maxSelectQueries = 1, maxRepeatedSelectExecutions = 1)
    void fetchJoinQueryShouldPassBudgetWithExtension() {
        int orderCount = memberQueryService.countOrdersWithFetchJoin();
        assertThat(orderCount).isEqualTo(6);
    }
}
```

여기서 가장 중요한 설계 포인트는 `scope` 경계입니다.

- `QueryGuardExtension`은 `@BeforeEach` 이후, `@Test` 직전에 scope를 연다
- 데이터 셋업 쿼리는 제외하고 검증 대상 메서드 실행 구간만 집계한다

이 경계를 명확히 두지 않으면 테스트 데이터 준비 쿼리가 섞여서 기준이 흔들리고, 결국 "실패해도 신뢰하지 않는 테스트"가 됩니다.

## 트레이드오프와 적용 범위

장점:

- JPA 중심 프로젝트에서 도입이 빠르다
- 정책 위반을 테스트 실패로 즉시 노출할 수 있다
- 반복 `SELECT` 패턴까지 함께 보여 분석 시작점이 명확해진다

비용:

- Hibernate 경로 밖의 쿼리는 별도 계측을 붙여야 한다
- 반복 `SELECT`는 "N+1 의심 신호"이지 항상 확정 진단은 아니다
- 멀티 ORM/멀티 데이터 접근 스택에서는 `DataSource` 레벨 계측이 더 적합할 수 있다

즉, 이 방식은 모든 팀에 대한 보편 해법이 아닙니다. 현재처럼 Spring Data JPA 중심 스택에서 "운영 전 회귀 차단"을 빠르게 만들 때 가장 효과적이었습니다.

## 정리

이번 작업의 결론은 아래 3가지입니다.

1. N+1 대응 우선순위를 "사후 디버깅"에서 "사전 회귀 차단"으로 변경
2. 초기 버전에서는 `StatementInspector`가 구현 복잡도 대비 효율 가장 좋음
3. 쿼리 카운터 성패는 집계 알고리즘보다 `scope` 경계 설계에서 갈림
