자바를 처음 접했을 때, 특히 면접 준비 과정에서 가장 먼저 들은 설명은 이랬습니다.

- primitive 타입은 call by value
- reference 타입은 call by reference

당시에는 크게 의심하지 않았습니다. 실제로 메서드 안에서 객체의 상태를 바꾸면 호출한 쪽에서도 값이 바뀌었고, 그 현상은 "참조로 전달된다"는 말로 충분히 설명되는 것처럼 보였습니다. 그렇게 이 설명은 별다른 검증 없이 지식으로 굳어졌습니다.

그러다 시간이 지나 다시 이 주제를 찾아보게 됐습니다. 이번에는 전혀 다른 설명이 보였습니다. 자바에는 call by reference가 없고, **항상 call by value만 존재한다**는 이야기였습니다.

더 찾아보니 이 주장은 개인적인 해석이나 일부 개발자의 의견이 아니었습니다. 자바 언어의 설계자인 **James Gosling**은 『The Java Programming Language』에서 자바의 메서드 인자 전달 방식을 명확히 pass-by-value 하나로 제한했다고 설명합니다. 객체를 전달하는 경우에도 객체 자체가 아니라 참조값(reference value)이 복사되어 전달된다는 점을 분명히 합니다. 자바 언어 명세인 Java Language Specification 역시 메서드 호출 시 인자는 값으로 전달된다고 정의하고 있습니다.

그래서 저 역시 이 전제를 받아들였습니다. “자바에는 call by reference가 없다”는 설명을 기준으로 기존에 알고 있던 개념을 다시 정리했습니다. 그런데 이후에도, 그리고 비교적 최근에도 면접에서 이런 질문을 받았습니다.

> "자바에서 call by value와 call by reference의 차이에 대해 아시나요?"

이 질문이 흥미로웠던 이유는, 제가 이 내용을 최근에야 다시 정리했기 때문입니다. 이미 공식적으로 정리된 개념임에도 불구하고, 왜 이 설명은 사라지지 않고 여전히 질문으로 반복되고 있는지 궁금했습니다. 단순히 잘못된 지식이 남아 있는 문제인지, 아니면 이 개념이 여전히 어떤 역할을 하고 있기 때문인지도 함께 궁금했습니다.

결국 제가 붙잡게 된 질문은 이것이었습니다.

> 왜 우리는 여전히 call by reference라는 말을 배우고,
또 그 개념이 사라지지 않은 채 면접 질문으로 반복되고 있는 걸까?

이 질문에 답하기 위해, 먼저 call by value와 call by reference가 무엇이 다른지를 간단한 코드로 확인하고, 그 차이가 실제로 언제 중요한지, 그리고 어떤 설명이 이해를 돕는 대신 비용을 만들고 있는지를 차례로 정리해보려 합니다.

---

## 기준 설정: 우리는 무엇을 설명하려고 이 개념을 사용하는가

call by value와 call by reference는 원래 **언어의 인자 전달 방식을 분류하기 위한 개념**입니다. 즉, “어떤 값이 전달되는가”가 아니라, **“전달 과정에서 무엇이 복사되고 무엇이 공유되는가”**를 설명하기 위한 분류 기준에 가깝습니다.

이 기준에서 보면 자바는 비교적 단순합니다. 메서드 호출 시 전달되는 것은 언제나 값이며, 객체의 경우에도 **객체 자체가 아니라 참조값이 복사되어 전달**됩니다. 이 사실만 놓고 보면 “자바는 call by value만 존재한다”는 설명은 충분히 정확합니다.

문제는 여기서 끝나지 않습니다.

현실에서 이 개념이 사용되는 맥락은, 언어 명세를 설명하는 자리보다 개발자가 코드를 이해하고 설명해야 하는 순간에 훨씬 더 자주 등장합니다. 대표적인 예가 다음과 같습니다.

- 메서드 안에서 객체를 변경하면, 왜 호출한 쪽의 객체 상태가 바뀌는지
- 어떤 메서드는 값을 바꾸지 않을 것처럼 보이는데, 실제로는 상태를 변경하는지
- 이 메서드 호출이 side-effect를 가지는지 한눈에 판단하기 어려운지

이 질문들에 답하려는 과정에서, call by reference라는 표현은 **설명 도구로서 매우 직관적**입니다. “참조로 전달된다”는 말은, 객체가 공유되고 있으며 변경이 외부에 영향을 미친다는 사실을 빠르게 전달해줍니다. 이 때문에 이 표현은 교육 과정과 면접 질문에서 오랫동안 살아남은 것으로 보입니다.

하지만 여기에는 중요한 전제가 빠져 있습니다. call by value / call by reference는 타입을 구분하기 위한 개념이 아니라, **호출 규칙을 설명하기 위한 개념**이라는 점입니다.

앞서 얘기한 것처럼 자바에서 흔히 들리는 설명은 이렇습니다.

- primitive 타입은 call by value
- reference 타입은 call by reference

이 설명은 이해를 돕기 위한 의도였을 수는 있지만, 개념적으로는 정확하지 않습니다. 자바는 reference 타입을 **별도의 전달 방식으로 취급하지 않습니다.** reference 역시 하나의 값이며, 다만 그 값이 **객체를 가리키는 값**일 뿐입니다.

이 지점에서 혼란이 발생합니다. 언어 차원에서는 하나의 규칙만 존재하는데, 설명 과정에서는 마치 두 가지 규칙이 존재하는 것처럼 전달됩니다. 그리고 그 결과, 개념은 단순해지는 대신 애매해집니다.

---

## 관찰/재현: 코드로 확인해보는 착시의 시작

call by value와 call by reference 논쟁이 반복되는 이유는 자바 코드가 **두 가지 전혀 다른 결과를 동시에 보여주기 때문**입니다. 먼저 가장 단순한 예제부터 보겠습니다.

### primitive 타입의 전달

```java
void change(int value) {
  value = 10;
}

int number = 5;
change(number);

// number == 5
```

이 결과는 직관적입니다. 메서드에 전달된 value는 number의 복사본이며, 메서드 내부에서 값을 바꿔도 호출한 쪽에는 아무 영향이 없습니다. 이 경우에는 누구도 혼란을 느끼지 않습니다. 문제는 다음 예제부터입니다.

### 객체를 전달했을 때

```java
void changeName(User user) {
  user.changeName("new");
}

User user = new User("old");
changeName(user);

// user.getName() == "new"
```

이번에는 결과가 다릅니다. 메서드 안에서 객체의 상태를 바꾸자, 호출한 쪽의 객체 상태도 함께 바뀌었습니다. 이 지점에서 많은 설명이 이렇게 이어집니다.

> “reference 타입은 call by reference입니다.”

하지만 이 결론은 너무 빠릅니다. 이 코드만으로는 **참조가 전달됐는지**, 아니면 **참조값이 복사됐는지**를 구분할 수 없습니다. 그래서 한 가지 예제를 더 봐야 합니다.

### 참조 자체를 바꿔보면

```java
void reassign(User user) {
  user = new User("other");
}

User user = new User("original");
reassign(user);

// user.getName() == "original"
```

여기서 중요한 점은, 메서드 안에서 user가 **새 객체를 가리키도록 변경되었음에도**, 호출한 쪽의 user에는 아무 영향이 없다는 사실입니다.

이 결과는 자바의 인자 전달 방식이 **call by reference가 아님을 결정적으로 보여줍니다.**

- call by reference라면

  → 참조 자체가 전달되므로, 재할당 결과가 호출자에게 반영되어야 한다

- 실제 자바의 동작은

  → 메서드에 참조값의 복사본이 전달되므로, 같은 객체의 상태 변경은 보이지만 재할당은 호출자에게 반영되지 않는다

핵심은 이것입니다.

**상태 변경이 보인다는 현상** 때문에 call by reference처럼 느껴지지만, 실제 전달 규칙은 끝까지 call by value입니다.

### 착시가 생기는 이유

혼란의 원인은 단순합니다. 사람들은 “무엇이 전달됐는가”보다, “무엇이 바뀌었는가”를 기준으로 판단합니다.

- primitive 값은 안 바뀐다 → value
- 객체 상태는 바뀐다 → reference

하지만 이 판단 기준은 **전달 방식이 아니라 결과에 기반한 추론**입니다. 자바는 처음부터 끝까지 동일한 규칙을 사용합니다. 다만 전달된 값이 **숫자이냐, 참조값이냐**에 따라 결과가 다르게 보일 뿐입니다. 이 지점에서 중요한 질문이 생깁니다.

> 그렇다면 굳이 call by reference라는 설명을 완전히 버려야 할까?

---

## 왜 call by reference라는 설명은 사라지지 않았는가

call by reference라는 표현이 오래 살아남은 이유는 단순합니다. 이 설명은 **빠르고, 직관적이며, 많은 상황에서 틀리지 않은 것처럼 보이기 때문**입니다.

### 이 설명이 제공하는 즉각적인 이점

- 객체를 전달했을 때 상태 변경이 외부에 반영되는 이유를 빠르게 설명할 수 있다
- “이 메서드는 객체를 바꾼다”는 위험 신호를 직관적으로 전달한다
- 초심자에게 side-effect의 존재를 즉시 인지시키는 데 효과적이다

교육 현장이나 면접에서 이 설명이 반복되는 이유도 여기에 있습니다. “참조로 전달된다”는 한 문장은, 객체 공유와 상태 변경이라는 핵심을 단숨에 전달합니다. 문제는 이 설명이 **어디까지 유효한지**를 함께 전달하지 않을 때 발생합니다.

### 설명의 유효 범위를 넘어서면 발생하는 비용

call by reference라는 표현은 **결과 설명에는 유리하지만, 규칙 설명에는 부적합**합니다. 이 차이를 구분하지 않으면 다음과 같은 비용이 쌓입니다.

#### 1. 규칙이 둘인 것처럼 오해하게 된다

- primitive → value
- reference → reference

이 설명은 자바에 **두 가지 인자 전달 규칙이 존재하는 것처럼** 보이게 만듭니다. 하지만 실제 규칙은 하나뿐입니다. 이 오해는 이후 개념 정합성을 계속 흔듭니다.

#### 2. 참조 값과 참조 대상의 구분이 흐려진다

“참조가 전달된다”는 표현은,

- 참조값이 복사되는 것인지
- 참조 자체가 공유되는 것인지  를 명확히 구분하지 못하게 만듭니다

이 결과는 코드 이해에서 바로 드러납니다.

- 왜 재할당은 반영되지 않는가
- 왜 불변 객체에서는 같은 설명이 통하지 않는가
- 왜 메서드 시그니처만 보고는 위험도를 판단하기 어려운가

이 질문들 앞에서 기존 설명은 더 이상 도움이 되지 않습니다.

---

## 실무에서는 이 개념을 어떻게 사용해야 할까

여기까지 보면 결론은 비교적 단순합니다. 자바에는 call by reference가 없고, 항상 call by value만 존재합니다. 그렇다면 이 개념은 실무에서 **어디에 써야 하고, 어디까지 써야 할까요.**

### 1. 메서드 설계에서 가장 중요한 기준은 “바뀌는가”입니다

실무에서 이 개념이 실제로 도움이 되는 순간은 **메서드 호출이 외부 상태를 변경하는지 판단해야 할 때**입니다.

```java
void update(User user) {
    user.changeName("new");
}
```

이 메서드는 호출자 입장에서 보면,

- 값을 넘겼지만
- 객체 상태가 바뀝니다

이때 중요한 질문은 “이게 call by reference냐”가 아니라,

> **이 메서드는 side-effect를 가지는가?**

라는 질문입니다.

call by reference라는 표현이 살아남은 이유도 여기에 있습니다. 이 표현은 **정확한 언어 설명**이라기보다는 “이 메서드는 객체를 건드린다”는 **경고용 표현**으로 기능해왔습니다.

### 2. 재할당이 반영되지 않는다는 사실은 설계 판단에 직접적인 영향을 줍니다

```java
void replace(User user) {
    user = new User("other");
}
```

이 코드가 호출자에게 영향을 주지 않는다는 사실은 다음과 같은 설계 판단으로 이어집니다.

- 메서드 내부에서 객체를 교체해도 호출자는 알 수 없다
- 객체 교체가 필요하다면 **반환값으로 명시해야 한다**
- 상태 변경과 객체 교체는 명확히 다른 책임이다

즉, call by value라는 규칙은 **메서드 설계 방식과 API 형태에 직접적인 제약**을 줍니다.

### 3. 그래서 이렇게 정리하는 것이 가장 현실적입니다

이 글에서의 정리는 이 정도가 적당합니다.

- **언어 규칙 설명**  → 자바는 항상 call by value이다

- **코드 이해·경고 목적의 설명**  → “객체를 넘기면 외부 상태에 영향이 간다”라고 말할 수 있다

문제는 이 둘을 **구분하지 않고 섞어 쓰는 것**입니다. 그 순간부터 설명은 편해질 수는 있어도, 정확해지지는 않습니다.

---

## 정리

> 자바에 call by reference가 없다는 사실보다 중요한 건,
**우리가 무엇을 설명하기 위해 그 말을 쓰고 있는지**입니다.

언어 규칙을 설명하는 자리라면 call by value만으로 충분합니다. 하지만 코드의 위험성, side-effect, 객체 공유를 빠르게 설명해야 하는 순간이라면 call by reference라는 표현이 여전히 등장하는 이유도 이해할 수 있습니다.

다만 그 표현이 **어디까지 유효한지**를 함께 설명하지 않는다면 이 개념은 이해를 돕는 도구가 아니라 혼란을 만드는 비용이 됩니다.
